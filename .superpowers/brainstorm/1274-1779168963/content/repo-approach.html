Template repo architecture

How should the 3 new sectors be structured on GitHub / Vercel?

A

3 separate repos (current pattern)

One repo per sector: biztemplate-trades, biztemplate-wellness, biztemplate-hospitality. Each is an independent Next.js app deployed to its own Vercel project.

Pros

  • Exactly mirrors existing stall pattern — zero new infra
  • Sectors can diverge freely over time
  • Independent deployments — one broken sector doesn't touch others

Cons

  • 3× the repos to maintain
  • Shared UI fixes must be applied 3 times
B

One shared repo, sector via config

Single biztemplate repo. A sector.config.ts file (written by the agent) controls which layout, sections, and components render. One Vercel project, one deployment.

Pros

  • Single codebase — fix once, all sectors benefit
  • Easier to add sector 4, 5, 6 later

Cons

  • All three clients share a single live URL — only shows the last deployed sector
  • Template complexity grows: conditional sections, sector branching in components
C

One shared codebase, 3 separate repos (monorepo fork)

Shared component library in a private package. Each sector forks it into its own repo and installs the library. Updates published once, consumed by all three.

Pros

  • Best of both: independent deployments + shared UI

Cons

  • Most complex setup — private npm package, versioning, publish step
  • Premature for 3 templates
